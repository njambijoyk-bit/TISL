All Cross-Module Links
⚡ triggers
payment-callback → loyalty-earn-order
When Payment callback confirms order as paid, Order::markAsPaid() triggers LoyaltyService::earnPointsForOrder()
↺ updates
payment-callback → payment-initiate
Callback syncs order.payment_status via syncOrderPaymentStatus(), which affects future initiate guards
⬡ depends-on
promo-validate → payment-initiate
Promo discount reduces order.total_kes, which affects the payment snapshot amount_still_owed
⚡ triggers
referral-complete → loyalty-grant-points
ReferralCodeUsage::complete() calls LoyaltyService::grantReferralCredit() / grantReferralPointBonus()
⬡ depends-on
credit-payment → payment-initiate
Credit account deductions reduce order total, affecting payment snapshot and balance calculations
⬡ depends-on
loyalty-spend-credit → payment-initiate
Store credit spent at checkout reduces total_kes, flowing into Payment::buildSnapshot()
⬡ depends-on
rec-populate → payment-index
ReconciliationPopulateService reads payments table to build reconciliation lines
⬡ depends-on
rec-populate → loyalty-index
ReconciliationPopulateService reads store_credit_transactions and loyalty_point_transactions
⬡ depends-on
rec-populate → credit-statement
ReconciliationPopulateService reads customer_credit_transactions
⬡ depends-on
loyalty-grant-points → referral-complete
Referral reward amount (referral_credit_amount) is read live from LoyaltySetting at reward time
⚡ triggers
promo-birthday → promo-validate
Generated birthday codes flow into the validation system at checkout
Key Insight — Cross-Module
Payment callback is the central trigger — it confirms orders, fires loyalty earn, and updates payment status. Store credit, promo discounts, and credit account deductions all feed into the payment snapshot before STK push. Reconciliation is a pure reader — it never writes to source ledgers, only pulls from them.

Key Insight — Payment
Daraja callback always returns Accepted to Safaricom — even on parse failures or duplicates. force_override bypasses balance/status guards but never bypasses cancelled/failed orders. Finance users can only see/cancel their own initiated payments. Retry links to previous_payment_id for full audit chain. Disputes: finance can raise but CANNOT resolve their own — admin only.

Key Insight — Loyalty
earnPointsForOrder is idempotent — checks loyalty_points_earned before writing, re-checks inside the transaction for race safety. Store credit capped at store_credit_max_pct (default 50%) of order total. Redemption minimum is 500 points. Expiry only removes expiring-type points — permanent points never expire. Sales reps can grant but NOT deduct points.
