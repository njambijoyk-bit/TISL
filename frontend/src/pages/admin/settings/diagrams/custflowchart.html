Key Insight — Auth
OAuth auto-accepts all policies silently — no user interaction. Email login triggers a policy re-acceptance check on every login before issuing the token. Trashed accounts are auto-restored on successful password verification. Referral code failures are always silent — never block registration.

Key Insight — Policies
Policy re-acceptance triggers on major version bump only (e.g. v1.x → v2.0). Critical policy disagreement flags the customer account AND notifies admin. Accepting a previously-disagreed critical policy clears all flags. Booking cancellation policy is checked per-booking, not at login.

Key Insight — Profile & Tier
Tier drives two things: discount_percentage on orders and loyalty_points_multiplier on earn. policy_flagged is the master block — it gates credit, loyalty redemption, and bookings simultaneously. CustomerTypeDiscount (individual vs corporate) applies an additional discount layer on top of tier discounts.

Key Insight — Credit
credit_used never goes below 0 on payment. Installment last item gets rounding remainder to avoid floating-point drift. Invoices are always draft first — admin must explicitly send them. Customer self-service excludes draft and void invoices. Overdue detection is dual-path: invoice due_date OR installment due_date, either triggers is_overdue flag.
⭐ Loyalty & Rewards Workflows
Points earning, reversal on cancel, store credit at checkout, redemption flow, and expiry command. 
flowchart TB
  EARN["Order marked paid\nearnPointsForOrder"] --> E1["loyalty_points_earned > 0? RETURN idempotent"]
  E1 --> E2["totalKes = order.total_kes"]
  E2 --> E3["totalKes <= 0? RETURN"]
  E3 --> E4["rate = LoyaltySetting points_per_100_kes default 1"]
  E4 --> E5["rawPoints = floor(totalKes/100) x rate"]
  E5 --> E6["rawPoints <= 0? RETURN"]
  E6 --> E7["multiplier = tier_benefits loyalty_points_multiplier default 1.0"]
  E7 --> E8["points = round(rawPoints x multiplier)"]
  E8 --> E9["expiryMonths = LoyaltySetting points_expiry_months"]
  E9 --> E10["DB transaction lockForUpdate on order"]
  E10 --> E11["Re-check loyalty_points_earned race condition guard"]
  E11 --> E12["writePointTransaction type=order_earn\nLock customer row\nnewBalance = loyalty_points + points\nUpdate customer.loyalty_points"]
  E12 --> E13["order.update loyalty_points_earned = points"]

  REVERSE["Order cancelled\nreversePointsForCancelledOrder"] --> R1["pointsToReverse = order.loyalty_points_earned"]
  R1 --> R2["<= 0? RETURN"]
  R2 --> R3["toDeduct = min(pointsToReverse, customer.loyalty_points)"]
  R3 --> R4["DB transaction\nwritePointTransaction type=order_cancel points=-toDeduct\norder.loyalty_points_earned = 0"]
  R4 --> R5["Allows re-earning if order restored and re-paid"]

  CHECKOUT_SC["applyStoreCreditToOrder\ncheckout"] --> SC1["available = customer.store_credit"]
  SC1 --> SC2["available <= 0? RETURN 0 0"]
  SC2 --> SC3["maxPct = LoyaltySetting store_credit_max_pct default 50"]
  SC3 --> SC4["maxAllowed = round(totalKes x maxPct/100, 2)"]
  SC4 --> SC5["deductionKes = min(requestedKes / available / totalKes / maxAllowed)"]
  SC5 --> SC6["deductionOrderCurrency = round(deductionKes / exchangeRate, 2)"]
  SC6 --> SC7["RETURN deductionKes / deductionOrderCurrency"]

  REDEEM["Customer or Admin redeem\nrule_id"] --> RD1["getRedemptionRules active only"]
  RD1 --> RD2["Rule NOT found: Exception"]
  RD1 --> RD3["required = rule.points_required"]
  RD3 --> RD4["required < minThreshold 500? Exception"]
  RD4 --> RD5["customer.loyalty_points < required? Exception"]
  RD5 --> RD6["DB transaction\nwritePointTransaction type=redemption points=-required"]
  RD6 --> RD7["valueKes > 0 AND type in cashback/voucher?"]
  RD7 --> RD8["writeCreditTransaction type=points_redemption amount=valueKes"]
  RD7 --> RD9["Gift type: no store credit granted"]
  RD8 --> RD10["RETURN: rule / points_used / credit_granted / message"]

  EXPIRY["ExpireLoyaltyPoints scheduled command"] --> EX1["LoyaltyPointTransaction due scope\npoint_type=expiring expires_at < now expired_at is null"]
  EX1 --> EX2["FOREACH tx"]
  EX2 --> EX3["customer missing? continue"]
  EX2 --> EX4["DB transaction\ntx.update expired_at=now\ntoRemove = min(tx.points, customer.loyalty_points)\nwritePointTransaction type=expiry points=-toRemove"]

Key Insight — Loyalty
earnPointsForOrder is idempotent — checks loyalty_points_earned before writing, and re-checks inside the transaction for race safety. Store credit is capped at 50% of order total by default (configurable). Redemption minimum is 500 points by default. Expiry only affects point_type=expiring transactions — permanent points never expire.

Key Insight — Algorithm
Scoring is NOT upsert — every run creates a new historical row, giving full score history over time. Segment rules are admin-configurable field/operator/value conditions that boost individual signals before final weighting. at_risk weights heavily toward sale items (35%) — dormant gets new products (30%). Pinned products always sort first regardless of score.
